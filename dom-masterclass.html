01 — Introduction

What is the DOM?

The Document Object Model is the browser's living, programmable representation of your HTML document. It's the bridge between your HTML structure and JavaScript's power.

The Three Pillars of Web

📄
HTML
Defines structure and content
🎨
CSS
Defines appearance and style
⚡
DOM + JS
Defines behavior and interaction

What JavaScript does with the DOM

🔍
Read
Access elements and their properties
✏️
Update
Change text, HTML, styles
➕
Create
Build new elements dynamically
🎧
Listen
Respond to user events
💡
Key Insight

The DOM is not your HTML file. It is a live, in-memory object representation that the browser creates from your HTML. When JavaScript modifies the DOM, your original HTML file stays unchanged — only the browser's representation updates.

How the DOM is created

PROCESS
1. Browser receives HTML from server (or local file)
2. HTML Parser reads the markup top-to-bottom
3. Parser creates Node objects for every element, text, comment
4. Nodes are organized into a tree (parent ↔ child relationships)
5. The tree is called the DOM — and JavaScript can now use it!

The entry point: always 'document'
document.getElementById('myId');
document.querySelector('.myClass');
document.createElement('div');

Terminology Glossary

Root — The topmost node. In HTML documents, document is the root.
Parent — A node that has child nodes beneath it. <ul> is parent of <li> elements.
Child — A node directly inside another node.
Sibling — Nodes at the same level with the same parent. <h1> and <p> are siblings.
Ancestor — Any node above on the path to root.
Descendant — Any node below in the tree.

03 — Node Types

Nodes vs Elements

Everything in the DOM is a Node. But not all nodes are elements. Understanding the difference is crucial.

Beginner
🎯
Key Distinction

An Element is a specific type of Node (one that comes from an HTML tag like <div>, <p>, etc.). But a Node can also be text content, a comment, the document itself, and more. Think of it like: all elements are nodes, but not all nodes are elements.

All Node Types — Click to Explore

ELEMENT_NODE (1) ATTRIBUTE_NODE (2) TEXT_NODE (3) COMMENT_NODE (8) DOCUMENT_NODE (9) DOCUMENT_TYPE_NODE (10)
👆 Click a node type to learn about it
JavaScript
// Every node has a nodeType property (a number)
const el = document.getElementById('myDiv');
console.log(el.nodeType);   // 1  → ELEMENT_NODE
console.log(el.nodeName);   // "DIV" (always uppercase)
console.log(el.nodeValue);  // null (only text nodes have values)

// Check the text inside an element
const firstChild = el.firstChild;  // Often a TEXT_NODE!
console.log(firstChild.nodeType);  // 3 → TEXT_NODE
console.log(firstChild.nodeValue); // "Hello" (the actual text)

// Node.ELEMENT_NODE is a constant = 1
if (node.nodeType === Node.ELEMENT_NODE) {
  console.log('This is an element!');
}

🧠 Quick Quiz

What is the nodeType of this: <!-- A comment -->?
A nodeType = 1 (ELEMENT_NODE)
B nodeType = 3 (TEXT_NODE)
C nodeType = 8 (COMMENT_NODE)
D nodeType = 9 (DOCUMENT_NODE)
✅ Correct! HTML comments have nodeType = 8. They are nodes in the tree but not elements, so querySelectorAll won't find them. You can access them via childNodes.
❌ Not quite. HTML comments are COMMENT_NODE with value 8. They exist in the DOM but are invisible to most element selectors.

04 — Selecting Elements

Finding Elements in the DOM

Before you can manipulate anything, you need to find it. JavaScript gives you several powerful tools for this.

BeginnerIntermediate
MethodReturnsSelects ByNotes
getElementById('id')Element | nullid attributeFastest; IDs must be unique
getElementsByClassName('cls')HTMLCollection (live)class nameCan pass multiple classes separated by space
getElementsByTagName('tag')HTMLCollection (live)tag nameUse '*' to get all elements
getElementsByName('name')NodeList (live)name attributeMainly used for form elements
JavaScript
// getElementById — returns a single element or null
const header = document.getElementById('main-header');

// getElementsByClassName — returns live HTMLCollection
const cards = document.getElementsByClassName('card');
console.log(cards.length); // count

// getElementsByTagName — by HTML tag
const allDivs = document.getElementsByTagName('div');
const everything = document.getElementsByTagName('*'); // all!
⭐
Preferred in modern JavaScript

querySelector and querySelectorAll use CSS selector syntax, making them incredibly flexible. They're the recommended approach for most use cases.

JavaScript
// querySelector — returns FIRST match or null
document.querySelector('#main-header');     // by ID
document.querySelector('.card');             // by class
document.querySelector('p.intro');           // p with class "intro"
document.querySelector('ul > li:first-child'); // CSS selectors!
document.querySelector('[data-id="42"]');     // attribute selector

// querySelectorAll — returns static NodeList of ALL matches
const cards = document.querySelectorAll('.card');
cards.forEach(card => console.log(card.textContent));

// Scoped queries — search within an element
const nav = document.querySelector('nav');
const navLinks = nav.querySelectorAll('a'); // only <a> inside nav

// Other useful methods
el.matches('.card.active'); // does el match this selector?
el.closest('.container');  // walk UP to find ancestor
⚠️
A Critical Difference: Live vs Static Collections

Live (HTMLCollection) — automatically updates when the DOM changes. getElementsByClassName, getElementsByTagName return these.
Static (NodeList) — a snapshot frozen at the time of the query. querySelectorAll returns this.

JavaScript — Traversal Reference
const el = document.querySelector('.item-b');

// ─── Going UP ───────────────────────────────────────
el.parentElement       // direct parent Element
el.parentNode          // direct parent Node (could be document fragment)
el.closest('.wrapper') // walk UP to find matching ancestor

// ─── Going DOWN ─────────────────────────────────────
el.children            // live HTMLCollection of element children only
el.childNodes          // live NodeList (includes text nodes, comments!)
el.firstElementChild   // first child element
el.lastElementChild    // last child element
el.firstChild          // first child NODE (often a text node with whitespace)

// ─── Going SIDEWAYS ─────────────────────────────────
el.nextElementSibling      // next sibling element
el.previousElementSibling  // previous sibling element
el.nextSibling             // next sibling NODE

// ─── Useful Checks ──────────────────────────────────
el.childElementCount   // number of element children
parent.contains(el)   // is el a descendant of parent?
⚠️
firstChild vs firstElementChild

This is a common trap! firstChild returns the first node, which is often a text node containing just whitespace (the line break and indentation between tags). firstElementChild skips text nodes and returns the first actual element. In modern code, prefer the Element variants.

06 — Content Manipulation

Reading & Changing Content

Three key properties let you read and change what's inside elements. Each has different behavior — understanding the differences matters.

BeginnerIntermediate

textContent

Gets/sets all text inside an element, including text in hidden elements. No HTML parsing — everything is treated as plain text. Safe from XSS.

innerText

Gets/sets visible rendered text only. Respects CSS styles (e.g., hidden elements excluded). Slower than textContent. Not available in some non-browser environments.

innerHTML

Gets/sets the HTML markup inside an element. Can create elements from strings. Dangerous if used with untrusted user input!

JavaScript
const el = document.querySelector('.box');
// Assume: <div class="box"><strong>Hello</strong> World</div>

el.textContent   // → "Hello World"
el.innerText     // → "Hello World" (same here)
el.innerHTML     // → "<strong>Hello</strong> World"

// Writing content
el.textContent = '<b>Bold?</b>'; // displays literally as text, safe!
el.innerHTML    = '<b>Bold!</b>'; // actually renders as bold HTML

// outerHTML — includes the element itself
el.outerHTML  // → '<div class="box">...</div>'
🚨
XSS — Cross-Site Scripting Warning

Never use innerHTML with content from users or external sources. An attacker can inject <script> tags or event handlers like <img onerror="steal()"> to run malicious code. Use textContent when inserting plain text, and sanitize HTML thoroughly before using innerHTML.

07 — Attributes vs Properties

Attributes & Properties

Attributes are defined in HTML. Properties are defined on DOM objects. They're related but not the same thing — and the difference is important.

Intermediate
💡
The Core Distinction

When the browser parses HTML, it creates DOM objects and initializes their properties from the HTML attributes. After that, attributes and properties can diverge! For example: changing input.value (property) reflects the current user-typed value, but getAttribute('value') still returns the original default.

JavaScript
// HTML: <input id="q" type="text" value="hello" disabled>
const input = document.getElementById('q');

// ─── Attribute methods ───────────────────────────────
input.getAttribute('value')    // "hello" — original HTML value
input.setAttribute('value', 'new') // sets the HTML attribute
input.removeAttribute('disabled') // re-enables the input
input.hasAttribute('disabled')   // true/false

// ─── Property access ────────────────────────────────
input.value      // current value (what user typed)
input.disabled   // true/false (boolean, not "disabled" string)
input.type       // "text"
input.id         // "q"

// ─── Custom data attributes ──────────────────────────
// HTML: <div data-user-id="42" data-role="admin">
const div = document.querySelector('div');
div.dataset.userId   // "42" (camelCase from kebab-case!)
div.dataset.role     // "admin"
div.dataset.newKey = 'value' // creates data-new-key attribute
📄 getAttribute() — Original HTML
value"hello"
disabled"disabled"
type"text"
⚡ .property — Live DOM State
.value"hello"
.disabledtrue
.type"text"

💡 Type in the input (first enable it), then refresh values. Notice how .value updates but getAttribute('value') stays at "hello"!

A user types "world" into <input value="hello">. What does getAttribute('value') return?
A "world" — it always reflects the current value
B "hello" — attributes preserve the initial HTML value
C null — typing removes the attribute
D "" — empty string after user interaction
✅ Correct! getAttribute('value') returns the original HTML attribute ("hello"). To get the current typed value, use the .value property instead.
❌ Attributes are frozen at their initial HTML values. Typing updates the .value property but leaves the value attribute untouched.

08 — Styling Elements

CSS from JavaScript

You can change how elements look from JavaScript — either by directly setting inline styles, or by toggling CSS classes (the better approach).

BeginnerIntermediate

Direct style property

JavaScript
// Read & write inline styles
el.style.color = 'red';
el.style.backgroundColor = '#fff'; // camelCase!
el.style.fontSize = '1.5rem';
el.style.display = 'none';   // hide
el.style.display = '';       // restore

// Get COMPUTED styles (includes CSS)
const styles = getComputedStyle(el);
styles.color       // "rgb(255, 0, 0)"
styles.fontSize    // "24px"

classList — the better way

JavaScript
el.classList.add('active');    // add class
el.classList.remove('hidden'); // remove class
el.classList.toggle('dark');   // add if absent, remove if present
el.classList.contains('btn');  // true/false
el.classList.replace('old','new'); // swap

// Multiple at once
el.classList.add('a', 'b', 'c');

// className (raw string)
el.className            // "card active featured"
el.className = 'card'  // REPLACES all classes
💡
Best Practice

Prefer classList over direct style manipulation. It keeps your styling in CSS (where it belongs), makes state management cleaner, and is more performant since CSS class changes can be GPU-accelerated and animated.

09 — Creating Elements

Building the DOM Dynamically

Rather than inserting raw HTML strings, you can construct elements programmatically — giving you full control and better security.

IntermediateAdvanced
JavaScript — Creating & Inserting
// ─── Create ────────────────────────────────────────
const btn = document.createElement('button');
btn.textContent = 'Click me';
btn.className = 'btn btn-primary';
btn.setAttribute('data-id', '42');

// ─── Insert ────────────────────────────────────────
parent.appendChild(btn);          // at the end
parent.prepend(btn);              // at the start
parent.insertBefore(btn, ref);    // before a ref element
ref.insertAdjacentElement('afterend', btn); // flexible!
// Positions: 'beforebegin', 'afterbegin', 'beforeend', 'afterend'

// ─── Remove & Replace ──────────────────────────────
btn.remove();                    // remove itself
btn.replaceWith(newEl);          // replace with another
parent.removeChild(btn);         // old way

// ─── Clone ─────────────────────────────────────────
const copy = btn.cloneNode(false); // shallow (element only)
const deep = btn.cloneNode(true);  // deep (element + all children)

⚡ DocumentFragment — Batch Insertion

🚀
Performance Tip

Each time you add an element to the DOM, the browser may perform a reflow/repaint. Adding 100 elements individually = 100 reflows. With DocumentFragment, you build everything in memory first, then insert it all at once = 1 reflow. This can be 10–50x faster!

JavaScript — DocumentFragment
// ❌ SLOW — 100 individual DOM insertions
for (let i = 0; i < 100; i++) {
  list.appendChild(document.createElement('li'));
}

// ✅ FAST — 1 DOM insertion using DocumentFragment
const frag = document.createDocumentFragment();
for (let i = 0; i < 100; i++) {
  const li = document.createElement('li');
  li.textContent = `Item ${i}`;
  frag.appendChild(li); // building in memory
}
list.appendChild(frag);   // ONE DOM touch — fast!
❌ Without Fragment (one by one)
✅ With DocumentFragment

10 — JavaScript Events

Making Pages Interactive

Events let your page respond to user actions. addEventListener is the modern way to attach event handlers — and it's packed with features.

BeginnerIntermediateAdvanced
JavaScript — addEventListener
// Basic syntax
element.addEventListener(eventType, handler, options);

// Simple click handler
const btn = document.querySelector('#myBtn');
btn.addEventListener('click', function(event) {
  console.log('Clicked!', event.target);
});

// Named function (so we can remove it later)
function handleClick(e) { console.log('click'); }
btn.addEventListener('click', handleClick);
btn.removeEventListener('click', handleClick); // same reference!

// Options object
btn.addEventListener('click', handleClick, {
  once: true,     // fire only once, then auto-remove
  capture: false, // use bubbling phase (default)
  passive: true   // won't call preventDefault() — better scroll perf
});

// The Event Object — key properties
e.target          // element that was CLICKED
e.currentTarget   // element the listener is ON
e.type            // "click", "keydown", etc.
e.preventDefault() // stop default behavior (link navigation, form submit)
e.stopPropagation() // stop bubbling/capturing
e.key             // for keyboard events: "Enter", "Escape", etc.
e.clientX, e.clientY // mouse position

11 — Event Delegation

One Listener, Many Elements

Instead of attaching listeners to every element, attach one listener to a parent and let bubbling do the work. This is one of the most important DOM patterns.

IntermediateAdvanced

❌ Without Delegation

JS — problematic
// 100 items = 100 listeners 😰
items.forEach(item => {
  item.addEventListener('click', handler);
});
// New items added later? No listener!

✅ With Delegation

JS — elegant
// 1 listener handles ALL items 🚀
list.addEventListener('click', (e) => {
  if (e.target.matches('.del-btn')) {
    e.target.closest('li').remove();
  }
});
🎯
Why Delegation Works

Because clicks on child elements bubble up to the parent. By checking e.target (what was actually clicked) in the parent's listener, you can handle events for all current and future children with just one listener. Use e.target.closest(selector) to safely handle clicks on nested elements inside list items.

12 — Advanced Concepts

State → DOM → CSS Pattern

The best DOM code follows a clear pattern: keep state in JavaScript, update the DOM to reflect state, and use CSS for visual presentation.

Advanced
JavaScript — State → DOM → CSS Pattern
// ─── 1. State (source of truth, JS object) ──────────
const state = {
  count: 0,
  theme: 'dark',
  isOpen: false
};

// ─── 2. Render function (State → DOM) ──────────────
function render() {
  countEl.textContent = state.count;
  document.body.dataset.theme = state.theme;
  menuEl.classList.toggle('open', state.isOpen);
}

// ─── 3. CSS handles visuals (DOM attribute → CSS) ──
/* [data-theme="dark"]  { background: #0a0e1a; } */
/* .menu.open           { transform: translateX(0); } */
/* .menu                { transform: translateX(-100%); } */

// ─── 4. Events only mutate state → call render ─────
btn.addEventListener('click', () => {
  state.count++;  // mutate state
  render();       // update DOM from state
});
🏗️
Why This Pattern Matters

This is the foundation of all modern frameworks (React, Vue, Svelte). By making state the single source of truth and re-rendering on state changes, your code becomes predictable, debuggable, and maintainable. React calls this "lifting state up." Always ask: "What is my state? How does my DOM reflect it?"

🔥 More Advanced Patterns

MutationObserver

Watch the DOM for changes — attribute changes, child additions/removals, text changes. Runs asynchronously (microtask queue).

JS
const observer = new MutationObserver((mutations) => {
  mutations.forEach(m => console.log(m));
});
observer.observe(target, {
  childList: true,
  attributes: true,
  subtree: true
});
observer.disconnect(); // stop observing

IntersectionObserver

Know when elements enter/leave the viewport. Perfect for lazy loading, infinite scroll, scroll-triggered animations.

JS
const io = new IntersectionObserver((entries) => {
  entries.forEach(e => {
    if (e.isIntersecting) {
      e.target.classList.add('visible');
    }
  });
}, { threshold: 0.2 });
io.observe(el);

Custom Events

Create and dispatch your own events to build loosely-coupled component communication.

JS
const event = new CustomEvent('userLogin', {
  detail: { userId: 42, name: 'Alice' },
  bubbles: true
});
document.dispatchEvent(event);

document.addEventListener('userLogin', e => {
  console.log(e.detail.name); // Alice
});

Debounce & Throttle

High-frequency events (scroll, resize, input) can fire 100s of times per second. Debounce/throttle prevent performance issues.

JS
// Debounce: wait until user stops typing
function debounce(fn, delay) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delay);
  };
}
input.addEventListener('input', debounce(search, 300));
What is the difference between e.target and e.currentTarget?
A They are always the same element
B e.target is where the listener is; currentTarget is what was clicked
C e.target is what was actually clicked; currentTarget is the element with the listener
D currentTarget only exists during capture phase
✅ Correct! e.target is the element that originally triggered the event (what the user clicked). e.currentTarget is always the element the listener was attached to. In delegation, currentTarget is the parent container, while target is the actual clicked child.
❌ e.target = the element the user actually interacted with. e.currentTarget = the element where addEventListener was called. They differ when delegation is used.
